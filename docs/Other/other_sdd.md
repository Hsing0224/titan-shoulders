# 規格驅動開發實戰：AI 時代的軟體開發新典範 - 以 GitHub Spec Kit 為例

Spec-Driven Development(SDD) 為一套方法論。<br />是一種把規格變成開發和新產物的軟體開發方法。

## 傳統開發 vs SDD

### 傳統開發流程

需求文件 -> 規格文件 -> 程式實作 -> 測試 -x-> 文件更新

> 程式碼才是唯一真相

### SDD 流程

需求輸入 -> 可執行規格 -> AI 生成測試與程式碼 -> 自動驗證與部署

> 程式碼由規格生成，更新規格即可再生實作，讓規格變成為可執行的文件

## GitHub Spec Kit

實現 SDD 方法論的完整工具組

### 核心理念

- 意圖導向開發
  開發者先定義意圖，做什麼(what)，再決定怎麼做(how)
- 豐富規格撰寫
  結合防呆機制和組織原則
- 多步驟的精練
  透過一步一步引導來產出高品質的 SDD
- **高度依賴先進 AI 模型對規格的解讀能力**
  先進的模型直接影響產出的品質。

### 基本流程

- Specify
  撰寫清晰可測的規格，定義使用者故事(情境)<br />
  寫不寫的對，是開發者的問題
- Clarify
  釐清模糊處，補足缺漏，透過 AI 對開發者靈魂拷問，幫助思考且釐清模糊的地方
- Plan
  給予技術規格與架構生成。可迭代使用。
- Tasks
  解析為可具體執行的任務，將任務轉換成任務清單
- Implement
  依照任務實作，驗證成果

:::note
如果產出不是你要的，把程式碼砍掉，修規格，修技術架構
:::

### 適用於 Spec Kit 的開發階段

#### 全新專案 (Greenfield)

非常適合 SDD 來實作

#### 現有專案 (Brownfield)

執行上較為困難，需要先寫出完整的 SDD，很難讓 AI 去理解整個程式碼的意圖

#### 創意探索

將 SDD 用於平行開發，產出不同的架構和框架

:::note
同規格每次跑不一定一樣
只要符合規格就是好規格
:::
