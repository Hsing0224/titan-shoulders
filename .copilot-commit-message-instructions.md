# Conventional Commits 1.0.0

## 概要

Conventional Commits 規範是一套輕量級的提交訊息約定。它提供了一套簡單的規則，讓開發者能夠建立明確的提交歷史記錄，進而更容易開發自動化工具。此規範與語義化版本控制（SemVer）相輔相成，透過提交訊息中的功能、修復和破壞性變更描述來協調版本變更。

## 提交訊息結構

提交訊息應遵循以下結構：

```
<類型>[可選的範圍]: <描述>

[可選的主體]

[可選的頁腳]
```

提交訊息包含以下結構元素，用於向函式庫的使用者傳達意圖：

### 核心類型

- **fix**：修復提交（`fix` 類型）用於修補程式碼中的錯誤（對應語義化版本控制中的 PATCH 版本號）
- **feat**：功能提交（`feat` 類型）用於向程式碼庫引入新功能（對應語義化版本控制中的 MINOR 版本號）
- **BREAKING CHANGE**：包含 `BREAKING CHANGE:` 頁腳或在類型/範圍後加上 `!` 的提交會引入破壞性 API 變更（對應語義化版本控制中的 MAJOR 版本號）。破壞性變更可以出現在任何類型的提交中

### 其他類型

除了 `fix` 和 `feat` 外，也允許其他類型。例如，基於 Angular 約定的 @commitlint/config-conventional 推薦使用：

- `build`：建置相關的變更
- `chore`：維護工作或依賴更新
- `ci`：持續整合設定的變更
- `docs`：文件更新
- `style`：程式碼風格調整（不影響功能的變更）
- `refactor`：重構程式碼
- `perf`：改善性能的變更
- `test`：測試相關的變更

> 注意：Conventional Commits 規範並未強制要求這些其他類型，且它們對語義化版本控制沒有隱含的影響（除非包含 `BREAKING CHANGE`）

### 範圍（Scope）

可以為提交的類型提供一個**範圍**，用來提供額外的上下文資訊，並使用括號括起來。
例如：`feat(parser): add ability to parse arrays`（`feat(解析器): 新增陣列解析能力`）

### 頁腳（Footer）

除了 `BREAKING CHANGE:` 以外，還可以提供其他頁腳，並遵循類似 Git trailer 格式的約定。

## 範例

### 包含描述和破壞性變更頁腳的提交訊息

```
feat: 允許提供配置物件擴展其他配置

BREAKING CHANGE: 配置檔案中的 `extends` 鍵現已用於擴展其他配置檔案
```

**說明**：此提交新增了配置擴展功能，但改變了 `extends` 鍵的使用方式，構成破壞性變更。

### 使用 `!` 標示破壞性變更的提交訊息

```
feat!: 在產品發貨時傳送電子郵件給客戶
```

**說明**：`!` 符號可快速吸引注意力，表示這是一個破壞性變更，無須額外的頁腳說明。

### 包含範圍和 `!` 的提交訊息

```
feat(api)!: 在產品發貨時傳送電子郵件給客戶
```

**說明**：指定了影響的範圍（API），並使用 `!` 標示破壞性變更。

### 同時包含 `!` 和 `BREAKING CHANGE` 頁腳的提交訊息

```
feat!: 停止支援 Node 6

BREAKING CHANGE: 使用 Node 6 不支援的 JavaScript 功能
```

**說明**：提供了額外的技術細節說明為何造成破壞性變更。

### 無主體的提交訊息

```
docs: 修正 CHANGELOG 中的拼寫錯誤
```

**說明**：簡單的文件修正，不需要額外的說明。

### 包含範圍的提交訊息

```
feat(lang): 新增波蘭語支援
```

**說明**：明確指出功能影響的模組或組件。

### 包含多段主體和多個頁腳的提交訊息

```
fix: 防止請求競速問題

引入請求 ID 和最新請求的參考。拒絕
接受來自非最新請求的回應。

移除原本用於緩解競速問題但現已過時的逾時機制。

Reviewed-by: Z
Refs: #123
```

**說明**：詳細說明問題、解決方案，並記錄相關的審查者和參考資訊。

## 詳細規範

下列文件中的關鍵字 "MUST"（必須）、"MUST NOT"（禁止）、"REQUIRED"（需要）、"SHALL"（應當）、"SHALL NOT"（不應當）、"SHOULD"（應該）、"SHOULD NOT"（不應該）、"RECOMMENDED"（推薦）、"MAY"（可以）和 "OPTIONAL"（可選）應按 RFC 2119 中的描述進行解釋。

### 提交訊息格式

1. 提交訊息**必須**以類型前綴開頭，該前綴由一個名詞組成（例如 `feat`、`fix` 等），後跟可選的範圍、可選的 `!`，以及必需的冒號和空格

2. `feat` 類型**必須**用於提交向應用或函式庫新增功能的情況

3. `fix` 類型**必須**用於提交代表應用的錯誤修復的情況

4. 可以在類型後面提供範圍。範圍**必須**由描述程式碼庫某個部分的名詞組成，並用括號括起來
   - 例如：`fix(parser):`

5. 描述**必須**緊接在類型/範圍前綴之後的冒號和空格後面。描述是程式碼變更的簡短摘要
   - 例如：`fix: 修復字串中包含多個空格時的陣列解析問題`

6. 可以在簡短描述之後提供更長的提交主體，以提供有關程式碼變更的額外上下文資訊。主體**必須**在描述後空一行開始

7. 提交主體是自由格式的，可以包含任意數量的換行符分隔的段落

8. 可以在主體後空一行提供一個或多個頁腳。每個頁腳**必須**由一個詞彙令牌組成，後跟 `: ` 或 ` #` 分隔符，再後跟字串值（靈感來自 Git trailer 約定）

9. 頁腳的令牌**必須**使用 `-` 代替空格字元（例如 `Acked-by`），這有助於區分頁腳部分和多段主體。`BREAKING CHANGE` 是一個例外，也可以用作令牌

10. 頁腳的值可以包含空格和換行符，解析**必須**在觀察到下一個有效的頁腳令牌/分隔符對時終止

11. 破壞性變更**必須**在提交的類型/範圍前綴或頁腳中指示

12. 如果作為頁腳包含，破壞性變更**必須**由大寫文本 `BREAKING CHANGE:` 組成，後跟冒號、空格和描述
    - 例如：`BREAKING CHANGE: 環境變數現在優先於配置檔案`

13. 如果在類型/範圍前綴中包含，破壞性變更**必須**由 `!` 標示，位於 `:` 之前。如果使用 `!`，可以省略頁腳部分中的 `BREAKING CHANGE:`，並使用提交描述來說明破壞性變更

14. 除 `feat` 和 `fix` 外，其他類型**可以**在提交訊息中使用
    - 例如：`docs: 更新參考文件`

15. 構成 Conventional Commits 的資訊單位實施者**禁止**將其視為區分大小寫的，但 `BREAKING CHANGE` **必須**為大寫

16. 用作頁腳中的令牌時，`BREAKING-CHANGE` 應與 `BREAKING CHANGE` 同義

## 為何使用 Conventional Commits？

使用 Conventional Commits 規範有以下優勢：

### 自動生成更新日誌（CHANGELOG）

根據提交類型和描述，可以自動生成版本發布的更新日誌，追蹤各版本的功能新增和修復。

### 自動判斷語義化版本號

根據登陸（合併）的提交類型自動判斷應進行的版本號變更：

- `fix` 類型 → PATCH 版本號升級
- `feat` 類型 → MINOR 版本號升級
- `BREAKING CHANGE` → MAJOR 版本號升級

### 向開發團隊和公眾溝通變更性質

清楚的提交訊息結構讓所有利益相關者（團隊成員、公眾、其他使用者）快速理解每次提交的目的和影響。

### 觸發建置和發佈流程

自動化工具可以根據提交類型和內容自動觸發相應的建置、測試和發佈流程。

### 簡化對專案的貢獻

結構化的提交歷史記錄使新貢獻者更容易理解專案的開發慣例，降低貢獻門檻，並鼓勵更多人參與開源專案。
